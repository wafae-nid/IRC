
# RFC 1459 vs RFC 2812

When learning IRC, a common question is:

> **If RFC 1459 already describes IRC, why do we use RFC 2812 for the `ft_irc` project?**

The answer is that **RFC 2812 is not a completely new protocol**. It is the **updated, clarified, and separated client-to-server portion of RFC 1459**.

---

# Timeline

```text
1993
│
├── RFC 1459
│   "Internet Relay Chat Protocol"
│
│   One RFC describing almost everything about IRC.
│
▼

2000
│
├── RFC 2810  IRC Architecture
├── RFC 2811  Channel Management
├── RFC 2812  Client Protocol
└── RFC 2813  Server Protocol
```

Instead of maintaining one large specification, the IRC protocol was reorganized into four RFCs, each focusing on a specific aspect of IRC.

---

# What RFC 1459 Contains

RFC 1459 is the **original IRC specification** published in 1993.

It contains everything needed to describe an IRC system:

* IRC architecture
* Clients
* Servers
* Channels
* IRC message format
* Client commands
* Numeric replies
* Server-to-server communication

Everything is described in a single document.

---

# Why Was RFC 1459 Split?

Between 1993 and 2000, IRC evolved considerably.

Developers found that one large RFC was difficult to maintain because it covered many independent topics.

Instead of rewriting IRC from scratch, the IETF reorganized the specification into four focused RFCs.

```text
RFC 1459
│
├── Architecture
├── Channels
├── Client Protocol
└── Server Protocol
        │
        │ Updated & Split
        ▼
RFC 2810  → Architecture
RFC 2811  → Channel Management
RFC 2812  → Client Protocol
RFC 2813  → Server Protocol
```

The protocol itself remained largely the same. The specification simply became more organized and easier to maintain.

---

# What RFC 2812 Contains

RFC 2812 focuses **only on communication between IRC clients and IRC servers**.

It specifies:

* Client registration (`PASS`, `NICK`, `USER`)
* Channel commands (`JOIN`, `PART`, `MODE`, `TOPIC`)
* Messaging (`PRIVMSG`, `NOTICE`)
* User commands (`WHO`, `WHOIS`, `QUIT`, ...)
* IRC message syntax
* Numeric replies
* Error replies

It intentionally does **not** describe:

* IRC architecture (RFC 2810)
* Detailed channel management (RFC 2811)
* Server-to-server communication (RFC 2813)

---

# What Did RFC 2812 Update?

RFC 2812 **did not redesign IRC**.

Most IRC commands already existed in RFC 1459.

For example, both RFCs define:

* `PASS`
* `NICK`
* `USER`
* `JOIN`
* `PART`
* `PRIVMSG`
* `NOTICE`
* `QUIT`
* `MODE`
* `TOPIC`

The basic IRC message format also remained the same:

```text
[:prefix] COMMAND parameters :trailing\r\n
```

Instead, RFC 2812 mainly improved the specification by:

## 1. Reorganizing the protocol

The client protocol was extracted into its own RFC, making it much easier to read and implement.

---

## 2. Clarifying command behavior

Many commands were explained more precisely.

RFC 2812 better defines:

* required parameters
* expected replies
* error conditions
* command syntax

This reduces ambiguities that existed in RFC 1459.

---

## 3. Updating numeric replies

The numeric replies and error codes were clarified and updated to reflect how IRC servers were actually behaving after several years of development.

---

## 4. Refining the specification

The IRC message syntax remained the same, but RFC 2812 explains it more clearly and removes ambiguities from the original specification.

---

## 5. Reflecting real-world IRC implementations

Between 1993 and 2000, IRC servers had matured.

RFC 2812 incorporates lessons learned from those implementations and standardizes many behaviors that had become common practice.

---

# Side-by-Side Comparison

| Feature                   | RFC 1459                 | RFC 2812                                                   |
| ------------------------- | ------------------------ | ---------------------------------------------------------- |
| Year                      | 1993                     | 2000                                                       |
| Purpose                   | Complete IRC protocol    | Client-to-server protocol                                  |
| Architecture              | ✅ Included               | ❌ Moved to RFC 2810                                        |
| Channels                  | ✅ Included               | Basic command behavior (full management moved to RFC 2811) |
| Client commands           | ✅ Original definitions   | ✅ Updated and clarified                                    |
| Message format            | ✅ Original specification | ✅ Clarified                                                |
| Numeric replies           | ✅ Original definitions   | ✅ Updated and clarified                                    |
| Server-to-server protocol | ✅ Included               | ❌ Moved to RFC 2813                                        |
| Primary use for `ft_irc`  | Historical reference     | ✅ Main implementation reference                            |

---

# Why Does the `ft_irc` Project Use RFC 2812?

The `ft_irc` project implements communication between an IRC client and an IRC server.

```text
          IRC Client
            (irssi)
               │
               │  RFC 2812 Messages
               ▼
         Your ft_irc Server
```

Your server does **not** communicate with other IRC servers.

Therefore:

* You do **not** need RFC 2813 (server-to-server protocol).
* You do **not** need most of RFC 2810 (architecture).
* You only need the client-server protocol, which is exactly what RFC 2812 specifies.

RFC 2812 is therefore a better implementation guide because it:

* focuses only on the protocol your project implements,
* is easier to navigate than RFC 1459,
* clarifies ambiguous behavior,
* reflects years of practical IRC implementation experience.

---

# Does `irssi` Use RFC 1459 or RFC 2812?

Historically, IRC clients were originally based on RFC 1459.

Modern IRC clients such as **irssi** implement the client protocol defined by RFC 2812, while also supporting additional IRCv3 extensions used by modern IRC networks.

---

# Which RFC Should You Read?

For the `ft_irc` project:

* **RFC 2812** should be your primary reference.
* **RFC 1459** is an excellent companion when you want historical context or additional explanations.

Reading both provides a deeper understanding of how IRC evolved and why the protocol looks the way it does today.

---

# Summary

Think of **RFC 1459** as the **original IRC textbook**.

It explained the entire IRC protocol in one document.

Think of **RFC 2812** as the **updated edition dedicated to the client-server protocol**.

It did not invent a new IRC protocol. Instead, it extracted, clarified, refined, and updated the client communication rules from RFC 1459.

Since the `ft_irc` project implements communication between IRC clients (such as **irssi**) and your server, **RFC 2812 is the primary specification you should follow**, while RFC 1459 remains a valuable historical and explanatory reference.

## Comparison

| RFC 1459                                                        | RFC 2812                                           |
| --------------------------------------------------------------- | -------------------------------------------------- |
| Original IRC specification (1993)                               | Updated client protocol specification (2000)       |
| Everything in one document                                      | Focuses only on the client ↔ server protocol       |
| Includes architecture, servers, channels, commands, and replies | Focuses on commands, replies, and message syntax   |
| Older wording and organization                                  | Clearer wording and better structure               |
| Useful for historical behavior and edge cases                   | Primary reference for implementing client commands |

---

## Which RFC should I use for `ft_irc`?

For the `ft_irc` project:

1. **Read RFC 2812 first.**

   * It is easier to understand.
   * It focuses on the commands you will implement.
   * It serves as the primary reference for client ↔ server communication.

2. **Consult RFC 1459 when needed.**

   * If RFC 2812 does not explain an edge case.
   * If you need historical context.
   * If you want to understand behavior followed by many traditional IRC servers.

---

## Summary

Think of the two RFCs this way:

* **RFC 1459** is the original IRC specification that explains nearly everything in one document.
* **RFC 2812** is a cleaner, reorganized specification dedicated to the client protocol.

For most of the `ft_irc` project, **RFC 2812 should be your primary reference**, while **RFC 1459 serves as a complementary reference for additional details and historical behavior.**
